# DECOM-PLAN — Decommissioning Plan

> [!note] Canonical CSV template — retirement-phase document
> **Canonical** template for the **Decommissioning Plan (DECOM-PLAN)** — the document that plans the controlled retirement of a validated computerized system (GAMP 5 retirement guidance — confirm §M-appendix number against the 2nd-edition index). It covers data migration/archival, GxP record retention, retrieval assurance, access revocation, shutdown sequence and verification. Executed before go-dark; closed by the Retirement Report.

> [!warning] GxP records must remain retrievable after shutdown
> EU Annex 11 §17 requires that archived records be readable for the full retention period. 21 CFR Part 11 §11.10(c) requires accurate and ready retrieval throughout the records retention period. The DECOM-PLAN must demonstrate how this obligation is met — including the technology, the archive location and a retrieval test — **before** the system is shut down.

> [!tip] Embedded usage rules
> 1. **Triggered by a retirement decision** — not necessarily preceded by a GXP-ASSESS (the system is existing/validated); link to [VR](VR.md) if one exists.
> 2. **Data Owner signature is mandatory** — accountability for GxP data does not end at shutdown.
> 3. **Archival verification before shutdown** — §5.3 must be marked complete before the shutdown steps in §6 are authorized.
> 4. **Rollback/abort criteria** (§8) must be defined in advance — shutdown may be aborted if archival verification fails.
> 5. **Closed by RETIRE-REPORT** — the Retirement Report documents actual execution against this plan.

## 1. Introduction

This **Decommissioning Plan (DECOM-PLAN)** defines the controlled retirement activities for the computerized system **`{{system_name}}`** (`{{system_id}}`), in accordance with GAMP 5 retirement guidance (confirm the §M-appendix number against the 2nd-edition index). It ensures that GxP records remain accessible for their full retention period after shutdown (EU Annex 11 §17; 21 CFR Part 11 §11.10(c)), and that all operational dependencies (access, licenses, integrations) are properly closed.

The plan is executed before go-dark. Actual execution results are documented in the Retirement Report.

## 6. Record retrieval assurance

> [!warning] Retrieval must be verified before shutdown
> EU Annex 11 §17 requires periodic checks to ensure data remains readable. A trial retrieval must be performed and passed **before** the system is shut down. The result is documented in the Retirement Report.

### 6.1 Pre-shutdown retrieval test

> [!note] Test IDs use the `-TC-` sub-prefix
> Retrieval/verification **tests** are identified `DECOM-PLAN-TC-NNN` (test-case namespace, per [requirement-id-scheme](../../docs/requirement-id-scheme.md)), distinct from the shutdown **step** IDs `DECOM-PLAN-NNN` in §8. The two counters are independent.

| Test | Description | Pass criterion | Result |
|---|---|---|---|
| DECOM-PLAN-TC-001 | Retrieve a representative sample of archived GxP records from `{{archival_destination}}` and confirm readability, integrity and metadata completeness | Records open correctly; content matches source; checksums pass; audit trail visible | |
| DECOM-PLAN-TC-002 | Confirm that the archived records can be retrieved by a user who does **not** have access to the decommissioned system | Retrieval succeeds without requiring the decommissioned system | |

### 6.2 Post-shutdown periodic retrieval tests

Frequency: `{{retrieval_test_frequency}}`

Periodic retrieval tests must be scheduled, executed and documented for the full retention period. Responsibility: Data Owner + Quality Unit. Results are recorded in the organization's periodic review or audit trail log.

## 10. Rollback and abort criteria

The decommissioning may be **aborted** if any of the following conditions arise before or during the shutdown sequence:

| Abort trigger | Action |
|---|---|
| Pre-shutdown retrieval test (DECOM-PLAN-TC-001/002) fails | Halt; investigate and resolve archive issue; re-test before proceeding |
| Archival integrity check fails (checksum mismatch or missing records) | Halt; remediate; re-verify before proceeding |
| Active GxP transactions discovered that have not been archived or migrated | Halt; complete disposition before proceeding |
| Data migration to replacement system not yet verified | Halt; do not proceed until migration is confirmed |
| Legal hold or regulatory inquiry received | Halt; consult Quality Unit and Legal before proceeding |

If aborted, the abort decision, reason and corrective actions must be documented and approved by the System Owner and Quality Unit before the plan is resumed.

## Canonical notes for implementers

> [!note] DECOM-PLAN is the companion to the VR, not a replacement
> The [Validation Report](VR.md) closes the validation project. The DECOM-PLAN opens the retirement project. Both may reference the same `system_id` — one entry, one exit.

> [!note] Archival format longevity
> EU Annex 11 §17 implies that the archive format must remain readable without the decommissioned application. Prefer open, non-proprietary formats (PDF/A, CSV, XML with schema) for long-term archival. If a proprietary format is unavoidable, document the migration path to readable formats and include it in periodic retrieval tests.

> [!tip] Retirement-plan approval
> Minimum signatories per GAMP 5 retirement guidance: System Owner + Data Owner + Quality Unit. The Data Owner signature is uniquely important here — they own GxP data accountability beyond system lifetime.

> [!warning] 21 CFR Part 11 §11.10(c) — ready retrieval obligation
> Records stored in a system that no longer exists must still be "readily retrievable." This means the archive system must be actively maintained, access procedures documented, and retrieval times within regulatory expectations. Verify with the Quality Unit what "ready" means for each record type.

## Instantiation flow (for AI agents)

> [!note] This section is NOT copied into the instance. It is guidance for the `gdd.decom` skill / orchestrator.

### Pre-flight

1. **Read `.gxp-dev.yaml`** — confirm `system_id` and locate the system's VR (`specs/VR.md`) if it exists.
2. **Read VR if it exists** — extract system name, intended use, validation history, GxP record types.
3. **Read RA-INIT if a retirement RA was produced** — contributes retirement-phase risks and mitigations.
4. **Read `templates/csv/DECOM-PLAN.md`** from the toolkit as the source template.

### Generation flow

1. **Inherit identity** from the VR or from `.gxp-dev.yaml` (name/id/system description).
2. **Scope and rationale** (§3): populate retirement rationale and replacement system if applicable.
3. **Record inventory** (§4): enumerate GxP record types, retention periods and disposition (archive/migrate/destroy).
4. **Data disposition** (§5): specify archival destination, migration destination (if any), destruction authorization.
5. **Retrieval assurance** (§6): populate retrieval test parameters and post-shutdown schedule.
6. **Access and contracts** (§7): enumerate user accounts, integrations, licenses to be closed.
7. **Decommissioning steps** (§8): complete the step table; add system-specific rows with sequential DECOM-PLAN-NNN IDs.
8. **Anti-hallucination**: use `[NEEDS CLARIFICATION: ...]` for any field that cannot be determined from available context.
9. **Output**: write `specs/DECOM-PLAN.md` (status: draft).
10. **Post-flight**: `validate-frontmatter.py` + `check-clarification-markers.py`.

### `status` transitions

```
draft ──[record inventory + archival plan + steps complete]──> in-review
in-review ──[System Owner + Data Owner + QU sign]──> approved
approved ──[execution complete → RETIRE-REPORT produced]──> superseded
```

### Downstream mapping

| Origin (DECOM-PLAN) | Destination | Rule |
