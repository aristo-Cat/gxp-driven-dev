# RTM — Requirements Traceability Matrix

> [!note] Canonical CSV template — DERIVED artifact
> **Canonical** template for the **Requirements Traceability Matrix (RTM)** — the matrix that demonstrates every requirement is specified, designed, and tested throughout the V-Model. Unlike the other templates, the RTM is **primarily auto-generated** by `skills/_scripts/generate-rtm.py` (GAMP 5 §M5 prefers tool-based traceability). This template documents the format produced + a manual authoring fallback.

> [!warning] Auto-generated — do not edit by hand
> If `generation_mode == auto-generated`, the RTM is regenerated on each script run. **Do not edit `specs/RTM.md` manually** — re-run the generator. Manually editing a derived RTM introduces divergence with the specs.

> [!tip] Embedded usage rules
> 1. **Forward + backward trace** — forward: requirement → spec → test; backward: test → spec → requirement → business need (GAMP 5 §D3.3.2 + §D5.5).
> 2. **Complete coverage** — every `URS-<CATEGORY>-NNN` with GxP=Y must trace to its test. Without traceability, the test does not satisfy validation (Annex 11 §9.5).
> 3. **0 dangling references** — an ID referenced that no document declares is a **bug** that blocks approval.
> 4. **Acceptable orphans** — IDs without downstream are OK only in the last layer (PQ-*) or GxP=N.
> 5. **Tool-based preferred** (GAMP 5 §M5) — automatic generation avoids the manual matrix becoming stale.

## 1. How it is generated

### 1.1 Auto-generation (recommended, GAMP 5 §M5 tool-based)

```bash
python skills/_scripts/generate-rtm.py --specs-dir specs --output specs/RTM.md
```

The script:
1. Traverses `specs/*.md`.
2. Extracts the canonical IDs `<DOC-TYPE>-<CATEGORY>-<NNN>` from each file.
3. Distinguishes **own IDs** (those the document declares, per its `template_id`) from **referenced IDs** (those from other documents it cites).
4. Builds the **forward trace** (upstream → downstream).
5. Detects **gaps/orphans** (no downstream; dangling references).
6. Writes `specs/RTM.md` with `type: derived`.

Exit codes: `0` OK · `1` no specs · `2` dangling references (bug to fix).

### 1.2 Manual fallback

For organizations without tooling: fill in the tables in §4-§7 by hand, following the same format. Cost: it becomes stale if not maintained (GAMP 5 §M5 advises against it).

## Canonical notes for implementers

> [!note] The RTM is derived, not a source
> Unlike URS/FS/etc., the RTM contains no original information: it is a **derived view** of the specs. That is why it is auto-generated. The source of truth is the specs; the RTM merely projects them.

> [!note] Consistency with generate-rtm.py
> The format of §3-§7 mirrors exactly what `generate-rtm.py` emits (Summary, per-file IDs, forward trace, gaps/orphans). If the script changes its output, this template is updated accordingly.

> [!tip] The RTM is key inspection evidence
> Annex 11 §6.5 + §9.5: untraceable test cases do not satisfy validation. An RTM with 0 dangling references and complete coverage of the GxP-critical requirements is direct inspection evidence.

## Instantiation flow (for AI agents)

> [!note] This section is NOT copied into the instance. It is guidance for the `gdd.trace.validate` skill.

### Flow (auto mode — recommended)

1. **Read `.gxp-dev.yaml`** from the consumer.
2. **Run** `python skills/_scripts/generate-rtm.py --specs-dir specs --output specs/RTM.md`.
3. **Check the exit code**: `2` = dangling references → report to the user as a blocking bug.
4. **Review gaps**: IDs without downstream that are NOT last layer / GxP=N → possible coverage holes.
5. **Output**: `specs/RTM.md` (type: derived). Do NOT edit by hand.

### Flow (manual mode — fallback)

1. Parse the IDs from each spec.
2. Fill in §4-§7 by hand following the format.
3. Set `generation_mode: manual` + `type: instance`.

### Mapping

| Origin (RTM) | Destination | Rule |
|---|---|---|
| Forward coverage | VR | The Validation Report cites the RTM as evidence |
| Dangling references | approval block | 0 dangling before approving specs |
| Coverage gaps | feedback to FS/OQ | GxP requirements without downstream = pending work |
