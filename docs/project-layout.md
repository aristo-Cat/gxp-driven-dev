# Project Layout — How a consumer project using `gxp-driven-dev` is structured

This document defines the **canonical layout** that any consumer repository should adopt when using `gxp-driven-dev` as its spec-driven development foundation — regardless of whether the consumer is a pharma organization, an indie founder, an R&D lab, or a consultancy.

It is the answer to: *"I want to develop software with AI agents using rigorous specs as authoritative context — how should my repo be ordered so the toolkit's templates fit naturally, the agent can read, instantiate, validate and cascade them across the lifecycle, and (if my domain is regulated) the same specs serve as compliance docs?"*

---

## Canonical structure

> Not all spec files are present in every project. Active templates are declared in `.gxp-dev.yaml` (`templates_active` list) and depend on `mode` (develop/validate/hybrid), `gamp_category` (1/3/4/5), and `profile` (pharma/medical-device/finance/aerospace/general).

## The `specs/` folder — purpose & rules

This is the **specifications folder** of the consumer project. It contains the instantiated artifacts of the V-Model lifecycle (or Agile equivalents), written as Markdown files using canonical templates from `gxp-driven-dev`.

**Rules:**

1. **One file per V-Model artifact.** Naming follows the template's acronym (`URS.md`, `FS.md`, `IQ.md`, `RA-INIT.md`, …).
2. **Active templates declared in `.gxp-dev.yaml`.** A skill can read the manifest and validate that the corresponding instance file exists.
3. **`RTM.md` is derived.** Generated by the `gdd.trace.validate` skill from the contents of `specs/*.md`; do not hand-edit.
4. **Instances are version-controlled (Git).** Each amendment to an approved spec follows the canonical rule: emit a new semver version, mark previous as `superseded`, never overwrite history.
5. **Anonymization not required here.** This is the consumer's own repository — corporate-specific references (system IDs, internal codes, site codes) are fine. The toolkit's anti-corporate-leak rule applies to the **toolkit's own source code and documentation**, not to repos that consume it.

## The `.gxp-dev.yaml` manifest

A small declarative file at the root of the consumer's repository. Skills read it to know which templates to instantiate, which presets to activate, and which toolkit version to assume.

### Full example

```yaml
# .gxp-dev.yaml — consumer manifest

# Toolkit version constraint
gxp_dev_version: ">=0.1.0,<1.0.0"

# Identity
project_id: "PROJ-2026-001"
project_name: "Lab Data Capture App"
language: en

# Mode (Decision #4)
mode: develop                  # develop | validate | hybrid

# Lifecycle
lifecycle: v-model             # v-model | agile | hybrid
gamp_category: 4               # 1 | 3 | 4 | 5 (required if profile == pharma)
lifecycle_phase: project       # concept | project | operation | retirement

# Profile (domain + rigor)
profile: pharma                # pharma | medical-device | finance | aerospace | nuclear | general
rigor_level: standard          # light | standard | strict | regulated

# ID scheme (Decision #5)
id_scheme: canonical           # canonical | custom
custom_alias:                  # only honored when id_scheme == custom
  URS: "REQ-USER-DOC"
  FS:  "SPEC-FUNC"
  # … map only the prefixes you want to alias

# Active presets
presets:
  part11_active: true          # activates URS-EREC + URS-ESIG sections (28 canonical reqs)
  annex11_active: true
  gdpr_active: false           # activates DPIA template
  annex22_active: false        # activates AISC template

# Active templates (subset of toolkit's 33)
templates_active:
  - VMP
  - GXP-ASSESS
  - VP
  - SUP-ASSESS
  - RA-INIT
  - URS
  - FS
  - DS
  - ADR
  - RA-DET
  - IQ
  - OQ
  - PQ
  - RTM
  - VR

# Outputs (Decision #7)
outputs:
  specs: true
  code: true                   # AI agent generates from approved specs
  tests: true                  # AI agent drafts test scaffolding from OQ/PQ
  compliance_bundle: true      # assembles compliance-bundle/ for regulated audits

# Hybrid breakdown (only when mode == hybrid)
hybrid_breakdown:
  vendor_components:
    - name: "LIMS core (commercial)"
      gamp_category: 4
      templates_active: [URS, CS, SUP-ASSESS, IQ, OQ, PQ, RTM, VR]
  custom_components:
    - name: "QC capture plugin"
      gamp_category: 5
      templates_active: [URS, FS, DS, ADR, API-SPEC, CR, UT-PLAN, IT-PLAN, IQ, OQ, PQ, RTM, VR, RN]
```

### Field reference

| Field | Type | Required | Origin decision | Description |
|---|---|---|---|---|
| `gxp_dev_version` | semver-range | ✅ | D1 | Toolkit version that this consumer expects |
| `project_id` | string | ✅ | — | Internal project identifier |
| `project_name` | string | ✅ | — | Human-readable project name |
| `language` | enum(en/es/de/…) | ✅ | — | Language of instantiated specs |
| `mode` | enum(develop/validate/hybrid) | ✅ | D4 | How the toolkit is used |
| `lifecycle` | enum(v-model/agile/hybrid) | ✅ | D7 | Shape of the SDLC |
| `gamp_category` | enum(1/3/4/5) | if profile==pharma | D3 | GAMP 5 §M4 categorization |
| `lifecycle_phase` | enum(concept/project/operation/retirement) | ✅ | — | GAMP 5 §3.1 |
| `id_scheme` | enum(canonical/custom) | ✅ | D5 | Requirement ID naming |
| `custom_alias` | map | if id_scheme==custom | D5 | Per-DOC-TYPE override |
| `profile` | enum(pharma/medical-device/finance/aerospace/nuclear/general) | ✅ | D2 | Domain profile (drives anchors) |
| `rigor_level` | enum(light/standard/strict/regulated) | ✅ | D2 | Severity of validation checks |
| `templates_active` | list of acronyms | ✅ | D3 | Subset of the 33 canonical templates |
| `presets.{part11,annex11,gdpr,annex22,…}` | bool | — | — | Activate canonical preset blocks |
| `outputs.{specs,code,tests,compliance_bundle}` | bool | ✅ | D7 | Which bundles to produce |
| `hybrid_breakdown` | object | if mode==hybrid | D4 | Vendor + custom components split |

### Why this manifest exists

- **`gdd.init` skill** reads it to know which templates to copy and pre-fill at consumer bootstrap.
- **`gdd.trace.validate` skill** reads `templates_active` to know which spec files must exist and be coherent.
- **CI gates** read `gxp_dev_version` to enforce that the consumer is using a supported toolkit version.
- The manifest makes the lifecycle state of the consumer project **machine-readable** (no need to grep through specs to find out "is this in development or in production?").

## How a developer uses the toolkit (intended workflow)

0. **Init** — run `/gdd.init` skill (interactive) or copy `.gxp-dev.yaml` example and fill identity + mode + profile + presets.
1. **Bootstrap** — `/gdd.init` creates `specs/` with stub spec files matching `templates_active`.
2. **Instantiate URS** — run `/gdd.urs.from-idea` interactive interview. Output: `specs/URS.md` with `status: draft`.
3. **Cascade** — for each downstream artifact, run the matching skill:
   - `/gdd.ra.from-urs` → `specs/RA-INIT.md`
   - `/gdd.fs.from-urs` → `specs/FS.md`
   - `/gdd.ra.detail.from-urs-fs` → `specs/RA-DET.md` (when FS approved)
   - `/gdd.tests.from-ra` → IQ/OQ/PQ drafts
4. **Iterate** — as the application is developed, requirements evolve. New requirements get new IDs; obsolete ones are *struck through* (never deleted). Use `/gdd.bump-version` to emit semver versions on approved specs.
5. **Validate** — run `/gdd.trace.validate` to ensure URS ↔ FS ↔ RA ↔ Tests coherence before promoting status to `approved`.
6. **Implement** — run `/gdd.implement-from-specs` to have the AI agent generate code using the approved specs as authoritative context. Review output, run tests.
7. **Operate** — once `status: validated`, the application enters operation. `PR.md` Periodic Reviews append entries on schedule. `CC.md` and `IR.md` log change controls and incidents.
8. **Bundle** — when an audit comes, run `/gdd.compliance-bundle` to assemble a static inspector-friendly bundle under `compliance-bundle/`.
9. **Decommission** — at end-of-life, instantiate `DECOM-PLAN.md` and (after retirement) `RETIRE-REPORT.md`.

## What this layout does NOT prescribe

- **The application's tech stack.** Python? Node? Go? .NET? Rust? Doesn't matter — `specs/` and `.gxp-dev.yaml` are agnostic to language.
- **The runtime architecture.** Monolith, microservices, serverless, mobile — the V-Model (or Agile) artifacts describe behavior, not deployment.
- **The corporate eQMS integration.** Once specs are approved, they may need to be uploaded to TrackWise / Veeva / MasterControl. That is consumer-specific and out of scope of the toolkit (roadmap Cat C).
- **Branding / theming of the spec documents.** The Markdown is the source of truth; PDF rendering for inspection is downstream and left to each organization.

## Open questions / future work

- **Bidirectional sync with corporate eQMS** — currently one-way (we produce specs, you upload them). A future skill could ingest changes back.
- **Multi-language instances** — the toolkit ships templates in English; future skills should support automatic translation while preserving canonical IDs.
- **Test automation linkage** — generating executable test stubs (e.g. `pytest`, `Playwright`, `Vitest`) from approved OQ/PQ test cases is a high-value future skill.
- **Compliance bundle differential mode** — for incremental audits, produce only what changed since the last bundle.
- **Per-component manifests** — for very large hybrid projects, `hybrid_breakdown.{vendor,custom}_components[*]` might need its own nested manifest. Deferred until real demand.
